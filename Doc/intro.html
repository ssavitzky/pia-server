Introducing the Platform for Information Applications

This document introduces the Platform for Information Applications (PIA).The PIA defines a new generation of document-oriented infoware with its reliance on XML to specify processing. The PIA framework divides an application into a core processing engine (called an agency) and task specific modules (called agents) which consist primarily of simple XML web pages.

The PIA functions as a web server that makes requests, processes documents, and handles inquiries on behalf of its "owner." This owner can be an individual making use of a web browser or it can be a network device.

An individual user might view the PIA as an agency employing electronic assistants, known as agents. These agents work on behalf of the user who owns the agency.

To use these agents, users must configure their web browser's proxy setting to point to their PIA agency. Once this is done, all requests for web documents are forwarded through the agency allowing agents to monitor the user's activities and retain useful information. (For example, the History agent maintains a searchable list of all documents ever visited by its owner.)

When the PIA acts on behalf of a device, such as a printer, it provides access to the functions of that device from any web browser. Agents can provide new features such as the ability to download, format, and print web pages directly. By presenting a web-based, easy-to-use interface, the agent can turn complicated, difficult to use computer peripherals into easy to use Information Applications.

The full potential of the PIA will be realized as users and developers build useful, task oriented agents. The PIA provides a solid foundation that enables rapid development and customization of agents. The PIA group has provided a number of example agents to demonstrate some possibilities. See the Agent Guide for a description of those agents that are currently available.

Working with PIA Agency

The PIA Agency is a program that manages a collection of software agents. Those agents operate on and communicate by means of messages called transactions. Transactions are of two types: requests and responses. A response is generated by an agent or external server in reply to some request, while a request is "unsolicited."

An agency can operate on behalf of an individual user, a group of users, or a peripheral device. Thus an agency can operate as:

  • a web server handing back documents in response to requests from a client such as a browser
  • a client requesting documents from other servers
  • a proxy passing requests and responses along to the appropriate servers and clients, and possibly operating on the documents in the process

This versatility opens the door to many possible applications:

  • Agents can "web-enable" a standard peripheral, such as a printer. Our web-printer can operate as a proxy, serving page previews in response to requests.
  • Agents can provide the basis for a document-processing "thin server." The web cache is an example of a thin server.
  • An agents can operate as your personal proxy, keeping track of your browsing history, performing filtering operations, and annotating documents. In addition, it can be a personal web server, making selected directories on your desktop machine available to others.

Outside the agency, requests and responses are transmitted using HTTP, a standard, widely used Internet protocol which is operating-system and programming-language independent.

An agency provides a means of examining, modifying and re-routing transactions among the various clients and servers. At times it operates as a client, as a server, as a cache, and as a proxy server depending upon the task it undertakes. As such, it provides a new platform for document-oriented computing.

How Agencies Work

Each transaction processed by an agency has a set of named features with corresponding values (mostly boolean) which are computed as needed. Each agent has a set of criteria that are matched against a transaction's features. Agents that match a transaction's criteria are allowed to act on it, that is perform whatever operation the agent is designed to do.

A transaction also has a queue of objects (mainly agents and transactions) called "satisfiers." After every agent has operated on a transaction, the agency attempts to satisfy the transaction by invoking the handle method of each object in the queue. If none of the methods return true, the agency performs a default action, generating an error transaction for an unsatisfied request or forwarding a response to its destination.

With the exception of a small number of built-in agents, everything that an agent does is accomplished by running an active document (for example, home.xh). Set-up is done by means of a document called initialize.xh. Further information on these files can be found in the document Authoring Agents.

Working with Agents

The PIA agency acts as a platform for computing using a collection of "agents." An agent's files are organized as a "sub-directory" of the agency's URL with each agent simply a collection of documents. These documents can include standard HTML tagging or make use of an extended set of tags which allows them to function as active documents.

This section provides a basic introduction to the use of these agents. For a more advanced example, see the Swifty Agent. Additional ideas and inspiration might come from a close look at the files, known as interforms, that make up the agents that ship with the PIA.

Agents and Interforms

That portion of the PIA that processes web documents is deemed an Information Agency because each document request passed through the agency is processed by one or more agents. Agents are persistent software objects running on the Information Agency. They perform one or more tasks on behalf of the agency's owner while maintaining state and context information.

State maintenance along with the agency framework allows agents to be more flexible, and much easier to generate and maintain than traditional web processes such as CGI scripts.

The basis for each Information Agency is a set of XML documents that can include one or more:

  • Standard HTML tags
  • Predefined PIA tags
  • User defined tags created using the interform language

When an agent receives a request for a document, it looks for the file that matches that request. Once the file is found, it is interpreted and the result returned.

For the end user, these files provide a form-based interface for interacting with the agent, hence they are called interforms. By convention, interform files are given the .xh filename extension, reflecting the fact that they contain an extended set of HTML tags.

The simplest agents might consist of one or more pages where each page is defined using a combination of HTML, XML and extended XML tags. There are also a number of predefined PIA tags available for use. See the Basic Tagset Manual for more on these tags.

Running an Information Agency

To run an Information Agency, the PIA must be installed at your site. If you install your own agency you can choose a custom set of agents as opposed to the default set.

Once it is installed, you can start the PIA by executing the command PIA_DIR/bin/pia, where PIA_DIR is the PIA home directory.

Viewing an Agent's Home Page

As soon as an agency is running, you can view that agency's home page at http://YOUR.HOST.NAME.HERE:8888/using any browser.

Notice the URL for this page. If your machine name is jaguar, you might see the following:

http://jaguar:8888/Admin/ROOT/index

Note that the URL for the agency home page has three parts:

  • http://jaguar:8888 represents the machine name and port number (your Information Agency).
  • Admin represents the name of the agent handling this request.
  • ROOT/index represents the path from that agent's root directory to the interform being displayed.

This URL constitutes a request to the Admin agent running on your local Information Agency. The Admin agent is the agent responsible for installing, removing, and listing other agents. It also manages miscellaneous tasks such as reporting bugs.

On your agency's home page, you should see a listing of all the agents currently running on your Information Agency. Click on one of the agent links to see the home page for that agent. By default, your agency is configured to start up with the following agents:

  • Admin is the main agent that manages all other agents
  • History maintains list of visited web pages
  • The Toolbar agencies
    • Localfile adds a control menu to pages served by your Information Agency
    • Remote adds a control menu to pages proxied through your agency
  • PIA serves files located in the PIA_DIR directory
  • ROOT serves files located in your local PIA_HOME directory (usually identified by~ )

These agents are further described in the Agent Guide.

Installing, Removing or Running Agents

The PIA home page provides interforms that can be used to perform basic management of agents including listing, adding and removing agents.

The agent list presents links to an index of all of the interforms for agents of a given type and to those agent's home pages.

For more information on agent installation see, Authoring Agents.

Program Invocation

The PIA Document Processor can be invoked stand-alone as a CGI or filter (formatter) or as part of the PIA. The command for invoking the document processor as a filter is:

process [option]... input_filename

The command-line options are:

-e
Do not define standard entities.
-h
Print a help string.
-o filename
Specify output file.
-p
Build a parse tree (which is output as a serialized Java object)
-t tagset-name
Specify tagset (default xhtml).
-s
Silent. Non-fatal error messages are suppressed.
-q
Quiet. Warnings and informational messages are suppressed.
-v
Verbose. Additional informational messages are output.
-d
Debugging. Information is output which may be of use for debugging the interpreter.

For example, a possible start up command is:

process -t xxml home.xh

The CGI version of the interform interpreter is: interform.cgi . Information is passed to the interpreter according to the Common Gateway Interface specification. The .cgi extension allows some servers (when suitably configured) to recognize it as a CGI script even if placed in a user's document directory.
